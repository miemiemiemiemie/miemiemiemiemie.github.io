LSA

Local Security Authority，用于身份认证，常见进程为lsass.exe

在系统启动的时候SSP会被加载到进程lsass.exe中

SSP是Security Support Provider，直译为安全支持提供者，又名Security Package.

简单的理解为SSP就是一个DLL，用来实现身份认证

默认共享

IPC$(Internet Process Connection)可以被理解为一种“专用管道”，可以在连接双方建立一条安全的通道，实现对远程计算机的访问。Windows NT/2000/XP提供了IPC$功能的同时，在初次安装系统时还打开了默认共享，即所有的逻辑共享(C$,D$,E$……)和系统目录(ADMIN$)共享。所有这些共享的目的，都是为了方便管理员的管理，只要有管理员权限就可以轻松访问这些

SPN

服务主体名称（SPN:Service Principal Names）是服务实例，可以将其理解为一个服务（比如 HTTP、MSSQL）的唯一标识符，服务在加入域中时是自动注册的。

当域控查询某一个服务的SPN时

如果该SPN注册在机器账户computers下，将会查询所有机器账户的servicePrincipalName属性，找到所对应的账户。

如果是注册在域用户账户users下，将会查询所有域用户账户的servicePrincipalName属性，找到其所对应的账户。

找到对应的账户之后，使用它的NTLM hash生成TGS。

域内的所有主机都是可以查询SPN的

域内的任何用户都是可以向域内的任何服务请求TGS

所以，域内的任何一台主机，都可以通过查询SPN，向域内的所有服务请求TGS，然后进行暴力破解，但是对于破解出的明文，只有域用户的是可以利用的，机器账户的不能用于远程连接，所以我们的关注点主要就在域用户下注册的SPN。

ACL

Access Control Link Windows

安全对象的ACL 所有的Windows对象都称为安全对象，文件、进程、 注册表、内存等 为每个安全对象赋予一个安全描述符（security desc riptor）

PAC

微软使用PAC来表示TGS中Client申明的域组。

PAC包含Client的User的SID、Group的SID。PAC决 定了Client的组属性，即决定了Client的权限

PAC为了保证自身的合法性，还包含2个签名，Key 为krbtgt的NTLM，签名的内容除了User SID、Group S ID外，还有其他部分 PAC作为TGT的一部分，是加密的，密钥为krbtgt的 NTLM

第一阶段放置在TGT中：

Client向KDC的AS模块发起认证请求，AS返回TGT 时，会根据Client所在的组，生成PAC，放在TGT中

第二阶段放置在TGS中：

Client使用TGT向KDC的TGS模块发起访问Server服务时，KDC的TGS模块首先解密TGT，并通过校验2个签名，以验证PAC的合法性。

如果通过验证，KDC的TGS模块用2个新的签名替代老的签名来保证PAC不被篡改。第一个签名的密钥为Server的NTLM，第二个密钥为Server与Client的临时会话密钥。重新签名后的PAC被放置在签发的访问票据TGS中， 使用Server的NTLM作为密钥

组策略的更新

客户端每90分钟主动获取一次组策略，检查组策略是否发生变更
客户端使用gpupdate/force强制立即检查组策略
参数可以分为主机和用户，gpupdate/force/target:computer

kerberos过程

  • KDC(Key Distribution Center):密钥分发中心，里面包含两个服务：AS和TGS
  • AS(Authentication Server):身份认证服务
  • TGS(Ticket Granting Server):票据授予服务,该服务提供的票据也称为 TGS 或者叫白银票据
  • TGT(Ticket Granting Ticket):由身份认证服务授予的票据(黄金票据)，用于身份认证，存储在内存，默认有效期为10小时

PTH攻击

理解

在NTLM认证时，若无指定，则使用当前NTLM进行认证，可使用mimikatz将获得的ntlmhash注入会话，登陆到其他域内主机

用法：在获得一台域内机器的权限后，通过pth得到其他域内机器的控制

条件：获得的域用户的NTLM HASH在两台（获得的机器和要攻击的）上相同且都在管理员组

实际运用场景：普通域用户在本地管理员组中，使用同一NTLMHASH管理域内机器

注意：在windows nt 5.x系统（Windows 2000/Windows XP/Windows Server 2003），管理员组账号的网络登陆都能获取系统权限

6.X系统（Windows Vista/Windows Server 2008/Windows 7/Windows Server 2008 R2 ）只有SID=500的才可以，其他管理员组成员远程登陆不可以获得完全管理员权限，只有交互式登陆才可以（UAC设置）即便没有打KB2871997，windows server 2008 r2依然阻止了RID非500本地管理员组账户的远程登陆（但是域管理员组的账户可以）

UAC

UAC通过一个注册表键值判断，若设为1，则SID500也不能远程最高权限

另一个注册表键值LocalAccountTokenFilterPolicy 若1，则所有管理员组可远程登陆最高权限

域用户若加入本地管理员组则不受以上约束（在非域环境受限多）

MS14-068

理解

Server收到Client发来的TGS后，要根据TGS中Client申明所在的域组，和Server上的ACL进行比对，然后决定给予Client什么样的资源访问权限。
微软使用PAC来表示TGS中Client申明的域组。PAC
（PrivilegeAttributeCertificate），特权属性书。PAC中有group的SID（决定权限）

注意的是，域SID是以域用户登录Win7域机器获得的，而漏洞利用则是在该域机器的本地账户下才能测试成功。

原因

1.KDC对PAC进行验证时，对于PAC尾部的签名算法， 虽然原理上规定必须是带有Key的签名算法才可以，但在实现上，却允许任意签名算法，只要客户端指定任意签名算法，KDC服务器就会使用指定的算法进行签名验证。

2.PAC没有被放在TGT中，放在其它地方。KDC仍然能够正确解析出没有放在TGT中的PAC信息 。（include-PAC标志设置为false两次后，就能导致PAC可以放在其它地方）

3.只要TGS_REQ按照刚才那两个要求设置，KDC服务器会做出令人吃惊的事情：它不仅会从Authenticator中取出来subkey把PAC信息解密并利用客户端设定的签名算法验证签名，同时将另外的TGT进行解密得到SessionKeya-kdc；在验证成功后，把解密的PAC信息的尾部，重新采用自身Server_key和KDC_key生成一个带Key的签名，把SessionKeya-kdc用subkey加密，从而组合成了一个新的TGT返回给Client-A

第五章 运输层

概述

进程间通信

运输层的一个很重要的功能就是 ‘复用’ 和 ‘分用’ 。应用层不同进程的报文通过不同的端口向下交到运输层，再往下就共用网络层提供的服务。

两个主机进行通信实际上就是两个主机中的应用进程互相通信。

应用进程之间的通信又称为端到端的通信。

“运输层提供应用进程间的逻辑通信”。

两个主要协议

用户数据报协议UDP 一条不可靠信道

传输控制协议TCP 全双工的可靠信道

端口

IP地址+端口号——唯一标识进程

三类端口
1熟知端口
2登记端口号——在IANA登记
3客户端口号或短暂端口号

UDP详解

主要特点

UDP 是无连接的，即发送数据之前不需要建立连接。

UDP 使用尽最大努力交付，即不保证可靠交付，同时也不使用拥塞控制。

UDP 是面向报文的。

UDP 没有拥塞控制，很适合多媒体通信的要求。

UDP 的首部开销小，只有 8 个字节。

UDP首部

用户数据报 UDP 有两个字段：数据字段和首部字段。首部字段有 8 个字节，由 4 个字段组成，每个字段都是两个字节。
在计算检验和时，临时把“伪首部”和 UDP 用户数据报连接在一起。伪首部仅仅是为了计算检验和。

UDP检验和是一个端到端的检验和。它由发送端计算，然后由接收端验证。其目的是为了发现UDP首部和数据在发送端到接收端之间发生的任何改动。

TCP详解

主要特点

TCP 是面向连接的运输层协议。

每一条 TCP 连接只能有两个端点(endpoint)，每一条 TCP 连接只能是点对点的（一对一）。

TCP 提供可靠交付的服务。

TCP 提供全双工通信。

套接字

TCP 把连接作为最基本的抽象。
每一条 TCP 连接有两个端点。
TCP 连接的端点不是主机，不是主机的IP 地址，不是应用进程，也不是运输层的协议端口。TCP 连接的端点叫做套接字(socket)或插口。
端口号拼接到(contatenated with) IP 地址即构成了套接字。

TCP 连接 ::= {socket1, socket2}
= {(IP1: port1), (IP2: port2)}

SOAP反序列化

来自LCTF2018

基础知识：

SOAP协议:SOAP 是一种简单的基于 XML 的协议，它使应用程序通过 HTTP 来交换信息。开启SOAP服务后可在phpinfo页面显示，在soap服务开启后soapclient就变成了php的内置类他的默认方法中包括
public __call ( string $function_name , array $arguments ) : mixed魔术方法，可在类调用不存在的方法时自动调用

soapclint用法如下
$client = new SoapClient($url, array("trace" => 1, "exception" => 0));
第一个参数为目标URL,第二个参数为一个数组，介绍如下
(mixed $wsdl [,array $options])

$option中包括execute_data,location,uri等参数，如果传入的第一个参数为$wsdl，这个参数控制是否为wsdl模式，若为wsdl模式，则在序列化之前就回对$url进行请求，如果值为NULL即非wsdl模式反序列化时就会对option中的uri进行远程soap请求。

e.g
<?php $a = new SoapClient(null, array('location' => "http://123.206.216.198:8887", 'uri' => "123")); echo serialize($a); ?>
自己复现注意：两边的服务器都要有SOAP服务才能完成整个过程
